# Cache Plan Logic for Recurring Queries - Design Document

## Overview
Implement a semantic cache for recurring action plans (e.g., "check spam") to skip 35B model inference on repetitive tasks, reducing latency and load on the Tesla P40 GPU.

## Goals
- Cache the *plan* (sequence of steps/tool calls) generated by the LLM for a given user query.
- On subsequent similar queries, retrieve the cached plan instead of re-running inference.
- Reduce inference cost and latency for repetitive tasks.
- Maintain correctness: invalidate cache when underlying context changes (time, user state, external data).

## Scope
- Cache key: semantic hash of the user query + relevant context (e.g., time window, user-specific flags).
- Cache value: serialized plan (JSON) produced by the LLM worker.
- Storage: lightweight, file-based (SQLite + optional pickle/blob) for simplicity and portability.
- Integration: transparent wrapper around the existing worker's planning step.

## Data Schema

### SQLite Table: `plan_cache`
| Column          | Type        | Description                                                                 |
|-----------------|-------------|-----------------------------------------------------------------------------|
| `cache_key`     | TEXT PRIMARY KEY | Hash (SHA-256) of the normalized query + context.                           |
| `plan_json`     | BLOB        | Serialized plan (JSON) produced by the LLM.                                 |
| `created_at`    | INTEGER     | Unix timestamp (seconds) when the entry was inserted.                       |
| `expires_at`    | INTEGER     | Unix timestamp (seconds) when the entry should be considered stale.         |
| `hit_count`     | INTEGER     | Number of times this cache entry has been reused (for metrics/eviction).    |
| `context_hash`  | TEXT        | Optional hash of additional context used (for debugging).                   |

### Plan JSON Structure (example)
```json
{
  "query": "original user query",
  "plan_id": "uuid or hash",
  "steps": [
    {
      "tool": "tool_name",
      "args": { ... },
      "description": "human readable"
    }
  ],
  "metadata": {
    "model": "qwen3.6-35b-a3b",
    "timestamp": 1734034567,
    "version": "1.0"
  }
}
```

## Hashing Mechanism
1. **Normalization**:
   - Lowercase the query string.
   - Remove extra whitespace.
   - Strip punctuation that does not affect meaning (optional, configurable).
   - Append context signals: current hour (for time-sensitive tasks), user ID if available, feature flags.
2. **Hashing**:
   - Compute SHA-256 of the normalized string.
   - Use the hex digest as the `cache_key`.

### Example Normalization (Python-like pseudocode)
```python
import hashlib, re, time
def normalize_query(query: str, context: dict) -> str:
    q = query.lower().strip()
    q = re.sub(r'\s+', ' ', q)  # collapse whitespace
    # optional: remove punctuation
    q = re.sub(r'[^\w\s]', '', q)
    # context: hour bucket (e.g., every 6 hours) to allow refresh
    hour_bucket = int(time.time() // 21600)  # 6-hour blocks
    context_str = f"|hour_bucket:{hour_bucket}"
    if context.get('user_id'):
        context_str += f"|user:{context['user_id']}"
    return q + context_str

def hash_key(normalized: str) -> str:
    return hashlib.sha256(normalized.encode('utf-8')).hexdigest()
```

## Expiration Policy
- **Time-Based TTL**: Each entry expires after a configurable time-to-live (default: 24 hours).
- **Context-Aware Invalidation**:
  - If the query includes explicit time references (e.g., "today", "now"), the hour bucket ensures separation.
  - For tasks dependent on external data (e.g., stock prices, news), the TTL should be short (e.g., 15 minutes) or the query should include a version/hash of that data.
- **LRU / Size Limit**: Optional maximum number of entries (e.g., 1000). When exceeded, evict least recently used (using `hit_count` and `created_at`).
- **Manual Invalidation**: Provide an API to purge specific keys or all entries (for updates, debugging).

## Integration Flow with Worker
1. **Worker receives query**.
2. **Before invoking the 35B model planner**:
   - Compute cache key from query + current context.
   - Look up `plan_cache` table.
   - If found and not expired (`expires_at > now`):
     - Deserialize `plan_json`.
     - Increment `hit_count`.
     - Return cached plan to the executor.
   - Else:
     - Proceed with normal LLM inference to generate plan.
     - Serialize plan to JSON.
     - Insert new row (or replace if key exists) with `created_at = now`, `expires_at = now + TTL`.
3. **Executor runs the plan** (cached or fresh) as usual.
4. **Metrics**: Log cache hits/misses for monitoring.

### Pseudocode Integration
```python
def get_or_create_plan(query, context):
    key = hash_key(normalize_query(query, context))
    row = db.fetchone("SELECT plan_json, expires_at FROM plan_cache WHERE cache_key = ?", (key,))
    now = time.time()
    if row and row['expires_at'] > now:
        plan = json.loads(row['plan_json'])
        db.execute("UPDATE plan_cache SET hit_count = hit_count + 1 WHERE cache_key = ?", (key,))
        return plan, True  # cache hit
    # cache miss
    plan = infer_plan_with_llm(query)  # existing worker logic
    plan_json = json.dumps(plan)
    expires_at = now + CACHE_TTL_SECONDS
    db.execute("""
        INSERT OR REPLACE INTO plan_cache (cache_key, plan_json, created_at, expires_at, hit_count, context_hash)
        VALUES (?, ?, ?, ?, ?, ?)
    """, (key, plan_json, now, expires_at, 0, hash_key(context)))
    return plan, False  # cache miss
```

## Configuration
- `CACHE_TTL_SECONDS`: default 86400 (24h), configurable via environment variable or config file.
- `MAX_CACHE_SIZE`: optional, default 1000 entries.
- `DB_PATH`: path to SQLite file (relative to project, e.g., `./cache/plans.db`).
- Context fields to include in hash: list of keys (e.g., `['user_id', 'session_id']`).

## Privacy & Public Code Considerations
- No absolute paths in code; use relative paths or environment variables.
- No hardcoded secrets; use placeholders (e.g., `os.environ.get('CACHE_DB_PATH', './cache/plans.db')`).
- Do not log raw user queries; only hashed keys for debugging.
- Provide a `.env.example` with dummy values.

## Testing Strategy
- Unit tests for normalization, hashing, DB operations.
- Integration test: run worker with a repetitive query, verify cache hit on second run.
- Performance test: measure latency reduction for cached vs. uncached plans.
- Expiration test: verify entry is invalidated after TTL.

## Diagram (Textual)
```
[User Query] --> [Normalize + Hash] --> {Cache Lookup}
                                   |-- Hit --> [Return Cached Plan] --> [Executor]
                                   |-- Miss --> [LLM Inference] --> [Store Plan] --> [Return Plan] --> [Executor]
```

## Open Questions
- Should we hash the *plan* itself for deduplication (different queries yielding same plan)? Not required for now.
- How to handle plan mutations (e.g., executor modifies plan)? Cache stores the original plan; executor works on a copy.
- Distributed cache? Single-node SQLite is sufficient for the current worker.
